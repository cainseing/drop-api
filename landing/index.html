Architecture

Drop encrypts data locally before it leaves your machine. It use AES-256-GCM with a unique key generated for every secret. The server only sees an encrypted blob.

Security Model

Every "Drop" consists of two components: the encrypted payload stored on our servers and the decryption key which resides solely in the token you share. Because tokens are never sent to the servers, we have Zero-Knowledge of your secrets content.

Primary Commands

drop [data]

Uploads data and returns a unique retrieval token. Content is encrypted before transmission.

Payload limit: 1MB per drop

drop get [token]

Fetch and decrypt the secret. The secret is destroyed immediately after retrieval if allowed reads have been exhausted.

drop purge [token]

Immediate server-side destruction. Permanently wipes the secret from storage without downloading or decrypting the payload.

Bypasses retrieval limits

Piping & Redirection

cat .env | drop

Stream any stdout directly into a secure drop. Perfect for logs, env files, or script outputs.

Payload limit: 1MB per drop

drop get [token] > .env

Download and redirect secrets directly into a file. The decrypted content never hits your clipboard or shell history.

Options

drop [data] --ttl 2

Override the default 5 minutes Time-to-Live.

Maximum window: 7 days

drop [data] --reads 5

Define a retrieval quota. The secret is purged automatically once the read count is exhausted.

Default: 1 (Strict Burn-on-Read)

Secrets are purged immediately upon reaching the retrieval limit or TTL expiration.
